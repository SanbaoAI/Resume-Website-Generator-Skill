# Design Taste Rules (Anti-Slop)

Premium taste standards for Stages 02–09. Adapted from design discipline in [taste-skill](https://github.com/Leonxlnx/taste-skill) — tuned for **personal resume / portfolio websites**, not generic SaaS landings.

**Read this file before Stage 04 and enforce it at Stage 09.**

---

## North Star

The site must feel **designed for this person**, not **generated by a model that defaults to the same portfolio template every time**.

Ask before every major decision: *"Would this look identical if I ran the same prompt for a different candidate?"* If yes, change something structural — layout family, palette family, type pairing, or section rhythm.

---

## 0. Design Read (Mandatory at Stage 04)

Before choosing colors, fonts, or layout variants, output a one-line **Design Read** in `design-strategy.md`:

> **Reading this as:** {portfolio kind} for {primary audience}, with a {vibe} language, leaning toward {aesthetic family}.

**Signals to read (from `candidate-analysis.md`):**

| Signal | What it picks |
|--------|----------------|
| Role (PM, engineer, designer, researcher, executive) | Density, motion, proof format |
| Brand keywords (precise, warm, bold, editorial, technical) | Palette temperature, type family |
| Audience (recruiter scan vs hiring manager deep-read vs client pitch) | Hero density, section order emphasis |
| Industry (B2B SaaS, creative, public sector, academia) | Trust vs expression balance |
| Existing assets (headshot, logo, published work) | Imagery strategy |

**Examples:**

- *Reading this as: senior PM portfolio for hiring managers, with a Linear-clean minimalist language, leaning toward cool neutral + single accent + restrained motion.*
- *Reading this as: creative technologist portfolio for design-conscious recruiters, with an editorial kinetic-type language, leaning toward asymmetric layout + scroll reveals.*
- *Reading this as: enterprise architect portfolio for procurement panels, with a trust-first language, leaning toward high density + static layout + explicit proof metrics.*

If the brief is genuinely ambiguous between two aesthetics, ask **one** clarifying question — never a list.

## 1. Three Dials (Set in Stage 04)

After the Design Read, set three dials in `design-strategy.md`. All layout, motion, and spacing decisions below are gated by these.

| Dial | Range | Meaning |
|------|-------|---------|
| **DESIGN_VARIANCE** | 1–10 | 1 = centered/symmetric · 10 = asymmetric/experimental |
| **MOTION_INTENSITY** | 1–10 | 1 = static · 10 = scroll-driven/cinematic |
| **VISUAL_DENSITY** | 1–10 | 1 = gallery-airy · 10 = information-packed |

**Baseline for resume sites:** `7 / 5 / 4` — asymmetric but readable, subtle motion, moderate content density.

### Resume Presets

| Candidate profile | VARIANCE | MOTION | DENSITY |
|-------------------|----------|--------|---------|
| Product / design leader (default) | 7 | 5 | 4 |
| Software engineer | 6 | 4 | 5 |
| Creative / brand / motion | 8 | 7 | 3 |
| Executive / strategist | 5 | 3 | 4 |
| Researcher / writer / academic | 6 | 3 | 3 |
| Data / analytics leader | 5 | 4 | 6 |
| Public sector / compliance-heavy | 3 | 2 | 5 |

### How dials change output

| Dial | Low | High |
|------|-----|------|
| VARIANCE | Centered hero, symmetric grids, predictable section stacks | Split hero, offset columns, mixed bento cells, intentional empty columns |
| MOTION | Hover/focus only | Staggered section reveals, stat count-up, nav scroll behavior |
| DENSITY | Fewer items per viewport, more whitespace | Stats, timelines, skill clusters visible without deep scroll |

**Motion contract:** If `MOTION_INTENSITY ≥ 5`, the page must actually animate (hero entrance + at least 2 section reveals). If you cannot ship working motion, lower the dial to 3 and ship static — never broken half-motion.

## 2. Anti-Default Discipline (LLM Slop)

Do **not** default to these — they are the most common AI portfolio tells:

| Slop pattern | Why it fails | Reach instead |
|--------------|--------------|---------------|
| AI purple/blue mesh gradient hero | Every generated site looks the same | Neutral base + one intentional accent |
| Inter + slate-900 everywhere | Generic "startup template" | Rotate display/body pairings per candidate |
| Three equal feature/project cards | Resume-template rhythm | Asymmetric project stack or bento with varied cell sizes |
| Uppercase eyebrow on every section | Templated section rhythm | Max 1 eyebrow per 3 sections (see §5) |
| Glassmorphism on everything | Style without hierarchy | Solid surfaces; glass only if design read calls for it |
| Warm beige + brass + espresso palette | "Premium artisan" default for every brief | Rotate palette families (see §4) |
| Serif display "because creative" | #1 tested AI tell | Default sans display; serif only when brief demands |
| Fake precise metrics (`92%`, `4.1×`) not in resume | Breaks trust | Only use numbers from resume or label as illustrative |
| Em dash (`—`) as copy flourish | Overused AI punctuation | Use comma, period, or colon; em dash banned in UI copy |
| "Download CV" primary CTA | Document viewer, not brand site | Contact / view work |
| Skill progress bars | LinkedIn clone | Capability clusters or proof-linked tags |
| Identical card shadow on every block | Box soup | Separate with space, hairline, or one elevation level |

## 3. Typography Taste

### Display font discipline

- **Default:** distinctive **sans** display — Syne, Outfit, Cabinet Grotesk, Satoshi, DM Sans, Plus Jakarta Sans, Geist-style grotesk
- **Discouraged as default:** Inter (acceptable only when brief explicitly asks Linear/neutral), Roboto, Arial
- **Serif display:** only when Design Read is editorial/luxury/publication **and** you can articulate why for this candidate
- **Banned as lazy defaults:** Fraunces, Instrument Serif (do not reach for these automatically)
- **Rotate pairings** — do not reuse the same Google Fonts pairing across consecutive projects

### Emphasis rule

Emphasize words inside headlines with **italic or bold of the same family**. Do not inject a random serif word into a sans headline (or vice versa).

### Overlines vs eyebrows

Section labels (overlines) create rhythm — but **eyebrow restraint applies** (§5). Not every section needs `text-transform: uppercase; letter-spacing: 0.1em`.

## 4. Color Taste

### One accent, locked

- Max **one** accent color for the entire page
- Accent saturation < 80% by default
- Once chosen, audit every CTA, link, stat highlight, and badge — no surprise second accent in section 6

### The "Lila Rule"

No automatic purple/violet glow, neon gradients, or blue-purple mesh backgrounds unless the brand keywords explicitly call for purple. Neutral bases (zinc, stone, slate) + one sharp accent (emerald, cobalt, burnt orange, deep rose, etc.).

### Palette rotation (mandatory)

Do not ship the same palette family for every candidate. Rotate among:

| Family | Character |
|--------|-----------|
| Cool minimal | Cool white + near-black + cobalt/cyan accent |
| Warm editorial | Warm off-white + charcoal + terracotta or sage |
| Dark tech | Rich dark + off-white + single vibrant pop |
| Monochrome + pop | Off-white + off-black + one saturated accent |
| Forest | Deep green + bone + amber accent |
| Cobalt + cream | Saturated blue against single neutral |

**Banned as default reach** (unless resume/brand explicitly names these): warm paper `#FAF7F1` + brass `#B08968` + espresso `#1A1714` — this is the premium-consumer AI default.

### Theme lock

Pick light OR dark for the whole page. No random light section sandwiched in a dark scroll (or vice versa) unless design strategy documents one deliberate theme switch.

## 5. Layout Taste

These apply **in addition to** [layout-system.md](layout-system.md) mandatory structure.

### Hero stack discipline

Max **4 text elements** in hero:

1. Eyebrow (optional — pick zero or one)
2. Name (visual anchor)
3. Tagline (h1)
4. Subhead + CTAs (subhead max ~20 words)

**Banned in hero:** feature bullet lists, "trusted by" logo walls, long paragraph bios, third CTA.

Hero must fit initial viewport on desktop — headline max 2 lines; reduce scale or cut copy, never overflow.

### Eyebrow restraint (critical)

Small uppercase tracking labels above section titles — **max 1 per 3 sections**.

If hero has an eyebrow, the next two sections cannot. Count before shipping.

### Section layout repetition ban

Each layout **family** may appear at most **once** per page:

- 3-column equal cards
- Split image + text zigzag (max **2 consecutive**)
- Full-width quote pull
- Bento grid
- Vertical timeline
- Stats strip

A page with 6+ sections must use at least **4 different layout families**.

### Bento cell count

Bento grids have exactly as many cells as content items. No empty placeholder cells.

### Cards and shadows

Use cards only when elevation communicates hierarchy. Prefer `border-top`, `divide-y`, or whitespace. Tint shadows to background hue — no pure black `drop-shadow` on light surfaces.

### Shape consistency

Pick one corner-radius system (sharp / soft 12–16px / pill buttons) and document it in `design-system.md`. Mixed random radii fail review.

## 6. Copy Taste

### Em dash ban

Do **not** use em dashes (`—`) in headlines, CTAs, eyebrows, or body copy. Use commas, periods, colons, or parentheses.

### Copy register

One voice per page — do not mix faux-poetic marketing lines with dry resume bullets in the same component.

### CTA discipline

- One label per intent sitewide ("联系我" OR "开始对话", not both)
- Primary CTA: 1–3 words ideal
- Button text must not wrap on desktop

### Metrics honesty

Hero stats and proof numbers must come from `resume-data.json` or be clearly generalized ("10+ 年" when resume says "2012–至今"). Never invent precision the candidate did not claim.

## 7. Motion Taste

Align with [animation.md](animation.md). Additional rules:

| MOTION | Required |
|--------|----------|
| 1–3 | `:hover`, `:focus-visible`, optional nav background fade |
| 4–6 | Hero entrance + scroll reveal on 2+ sections + stat count-up |
| 7–10 | Above + staggered children + optional parallax on decor only |

Every animation needs a **purpose** (hierarchy, feedback, narrative sequence). "It looked cool" is not a purpose.

Always respect `prefers-reduced-motion: reduce` — disable non-essential motion.

## 8. Pre-Flight Checklist (Stage 09 — Critical if failed)

Run mechanically before scoring. Any fail → **Critical** issue, fix before delivery.

| # | Check | Pass criteria |
|---|-------|---------------|
| 1 | Design Read exists | `design-strategy.md` has one-line Design Read + three dials |
| 2 | Name-first hero | Name largest element; 4 stats from real proof |
| 3 | Floating pill nav | Not edge-to-edge sticky bar |
| 4 | Eyebrow count | ≤ ceil(sectionCount / 3) uppercase tracking labels |
| 5 | Layout families | ≥ 4 distinct families on 6+ section pages |
| 6 | Accent lock | One accent hue sitewide |
| 7 | Slop scan | No skill bars, purple mesh hero, em dashes in copy |
| 8 | Font pairing | Not default Inter+slate unless documented in strategy |
| 9 | CTA intent | One contact label, no "Download CV" primary |
| 10 | Theme lock | No mid-page light/dark flip |
| 11 | Motion match | Motion present iff dial ≥ 5; reduced-motion respected |
| 12 | Content accuracy | Rendered text matches resume-data.json |

---

## 9. Differentiation Mandate

Every `design-strategy.md` must include a **Differentiation Statement** answering:

1. What layout/color/type choices are **specific to this candidate**?
2. Which slop patterns from §2 are explicitly rejected for this project?
3. If two candidates in the same role used this skill, what would make their sites visibly different?

If you cannot answer (3), revisit Stage 04 before Stage 05.
